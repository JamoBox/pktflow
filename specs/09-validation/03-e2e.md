# 09.3 — End-to-end stream tests

> Task: [09 Validation](README.md) · Depends on: 08.5 (JSON surface), 09.2 · PRD: §7 aggregation testability, §8 "stream fidelity"

## Goal
Whole-pipeline assertions: fixture in → expected stream tree out, plus the success-metric
parity check against an established reference tool's flow accounting.

## Specification

**Tree assertions (library level):** for every 09.2 synthetic fixture, run
`MockSource → dissect → Aggregator::ingest → finish → snapshot` and diff against its
`ExpectedStreams`:

```rust
pub struct ExpectedStream {
    pub protocol: &'static str,
    pub endpoints: (&'static str, &'static str),   // rendered form, canonical order
    pub packets: (u64, u64), pub bytes: (u64, u64), // per direction
    pub state: Option<&'static str>, pub closed: Option<CloseReason>,
    pub rollups: Vec<(&'static str, ExpectedRollup)>,
    pub children: Vec<ExpectedStream>,              // the tree shape IS the assertion
}
```

Diff output on mismatch renders both trees side by side — stream-test failures must be
readable or nobody will maintain them.

**CLI-level goldens:** the same fixtures through the real binary (`assert_cmd`), asserting
`--format json` against goldens (08.5 schema) and text views against 08.2/08.3 goldens.

**Reference parity (PRD §8 "stream fidelity"):** for each *real* capture, a comparison
harness (`crates/pktflow-cli/tests/reference_parity.rs`) asserts pktflow's per-layer
packet/byte accounting matches tshark's (dev/CI dependency only — never a runtime
dependency). Implemented as exact `tshark -Y <filter> -T fields -e frame.len` sums rather
than parsing `-z conv,X`'s text report: that report rounds byte counts to `kB`/`MB` past
~1000 bytes, which is fine for a human and useless for an exact assertion. Known model
differences are documented adjustments in the harness: `dhcp_dora.pcap`'s client address
changes mid-exchange, so tshark's address-based grouping sees two conversations where
pktflow's DHCP-aware flow key sees one (summed packets/bytes still match exactly, so counts
aren't compared — only sums); `vxlan_overlay.pcap` needs its tunneled inner layer's
contribution added explicitly, since `at_layer_merged` (D10) counts a tunneled frame once
per stream it belongs to (outer + inner) the same way tshark's own conv report
double-counts across its outer/inner rows, but a plain filtered frame-length sum only
counts each frame once. Unexplained deviation = failure.

**Determinism e2e (PRD §7):** every fixture run twice in-process + twice via CLI ⇒ identical
snapshots and byte-identical JSON (the 00.3 smoke, now over the full corpus).

## Acceptance criteria
- [x] All synthetic fixtures pass tree assertions; failure output human-verified readable.
- [x] tshark parity green on all real captures, adjustments documented inline.
- [x] The QUIC "honest-unknowns" capture: unknown payloads land in `stop_classes.
      unknown_payload` + UDP stream `opaque_bytes`; **zero** streams of any protocol beyond
      UDP (the §8 no-phantom metric, measured on real traffic).
- [x] Determinism suite green on both CI OSes.
